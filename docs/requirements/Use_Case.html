    1. Name: Present Cart Home Page

      • Actors:
        User

      • Brief Description
        Home is the main page for the shopping cart. The presentation of the home page can be modified by configuration settings (ie. default display of the home page may be a browse category listing, while one admin may want to change this to search product). The home page may also take into account user personalization if that becomes available at a later time.

      • Flow of Events
        Basic Flow
        User Action System Response
        1. User enters cart home page 2. System determines home page layout and displays.

      • Special Requirements
        - The home page would be a prime candidate for personalization, this should be taken into account.
        - The home page display should be cached. It should not be parsed everytime it is displayed until personalization is added.

      • Pre-Conditions
        none

      • Post-Conditions
        The home page is properly displayed.

      • Cross Reference
        none

      • Assumptions
        none

    2. Name: Browse Product Catalog

      • Actors:
        User

      • Brief Description
        A user can browse the product catalog by categories that categorize products into logical groupings. This allows a user to to locate products easily by following pre-defined trails of categories.

      • Flow of Events
        Basic Flow
        User Action System Response
        1. User selects the category with which to browse the catalog. 2. System displays the category template page, dynamically generating information about the selected category.
        3. User continues to choose sub-categories until individual products of interest are displayed. 4. System displays the product list template page.

      • Special Requirements
        - A category may contain sub-categories as well as products

      • Pre-Conditions
        none

      • Post-Conditions
        none

      • Cross Reference
        -         View Product Listing Use Case

      • Assumptions
        - The categories will be setup before hand through administrative interface or database interface.
        - Products will be assigned to categories before hand through administrative interface or database interface.

    3. Name: View Product Listing

      • Actors:
        User

      • Brief Description
        A user may get to this view in multiple ways (search, browse...). This view presents the user with a list of products that will are available for purchase.

      • Flow of Events
        Basic Flow
        User Action System Response
        1. User navigates to product listing page via search or browse interface. 2. System looks up products based on inputs.

        		 3. System displays list of products, dynamically generating information about the products within the selected category. The details displayed for the product listing are determined through system configuration.

        Alternative Flow
        User Action System Response

        		 2.1 System looks up products based on inputs, but finds not products.

        		 2.2 System displays error page with relevant error message.

      • Special Requirements
        - Details to display for product listing page must be configurable at deploy time.
        - The option to view detail on each product is given.
        - The option to add the product to a users shopping cart is available from this view.

      • Pre-Conditions
        none

      • Post-Conditions
        none

      • Cross Reference
        -         Browse Product Catalog Use Case
        - View Product Detail Use Case

      • Assumptions
        none
      Top

    5. Name: Add To Cart

      • Actors:
        User

      • Brief Description
        Allows a user to store the products they are interested in purchasing in a virtual basket or cart.

      • Flow of Events
        Basic Flow
        User Action System Response
        1. User selects an individual product to add to their cart. 2. System verifies the existance of a cart and a session

        		3. System adds the product to the users cart

        		4. System redirects to the Manage Cart Use Case

        Alternative Flow
        User Action System Response

        		 2.1 System does not find a cart or session

        		2.2 System creates a cart and a session

      • Special Requirements
        - System should increase the quantity by 1 of products already in cart.

      • Pre-Conditions
        none

      • Post-Conditions
        The product has been added to the users cart

      • Cross Reference
        -         View Product Detail Use Case
        - Manage Cart

      • Assumptions
        none

    6. Name: Manage Cart

      • Actors:
        User

      • Brief Description
        Allows a user to change quanities of products in cart. User can remove products from cart.

      • Flow of Events
        Basic Flow
        User Action System Response
        1. User selects to view cart or user has been redirected to cart view after a cart change 2. System verifies there is a cart for the user

        		3. System makes the user specified changes to cart.

        		4. System displays a summary of the products in the users cart. System displays line item subtotals and overall cart subtotals

        Alternative Flow

      • Special Requirements
        User should be able to select item in cart and view product details

      • Pre-Conditions
        none

      • Post-Conditions
        none

      • Cross Reference
        -         View Product Details
        - Add To Cart

    7. Name: Check Out Cart

      • Actors:
        User

      • Brief Description
        The process a user goes through to order the items in their cart. They will need to enter billing and shipping info. Their payment might need to be authorized. The availability of items may need to be checked. Appropriate taxes will be added.

      • Flow of Events
        Basic Flow
        User Action System Response
        1. User selects Check Out. 2. System asks for shipping info
        Fields:
        Name
        Address
        Suite/Apt/Box/etc
        SCity
        State
        Zipcode
        Country
        3. User enters shipping info 4. System verifies shipping info

        		5. System stores shipping info

        		6. System asks for billing info
        Fields:
        Name on Card
        Card Type
        Card Number
        Card Expiration
        Phone
        Email
        7. User enters billing info 8. System verifies billing info

        		9. System stores billing info

        		10. System shows cart summary(with subtotals and totals), shipping info, and billing info
        11. User verifies the order is correct and confirms 12. System authorizes payment.

        		13. System sends order to fulfillment

        		14. System emails user reciept.

        		15. System displays reciept.

        Alternative Flow
        User Action System Response

        		 4.1 System had a problem with the shipping info

        		4.2 System redirects to 2 with relevant errors

        Alternative Flow
        User Action System Response

        		 8.1 System had a problem with the billing info

        		8.2 System redirects to 6 with relevant errors

        Alternative Flow
        User Action System Response

        		12.1 System could not authorize payment

        		12.2 System redirects to 6 with relevant errors

      • Special Requirements
        - All data should be securely transmitted between client and server for this entire process.
        - Configurable Process Flow - A way to move the user / cart session data from one server to another.

      • Pre-Conditions
        - A user has products in their cart.

      • Post-Conditions
        none

      • Cross Reference
        -         View Receipt
        - Sign On User

      • Assumptions
        none
      Top

    8. Name: View Reciept

      • Actors:
        User

      • Brief Description
        Allows a user to view what they ordered, how it will be shipped, and how much they paid in taxes.

      • Flow of Events
        Basic Flow
        User Action System Response
        1. A user completes an order. 2. The system displays a receipt for the user.
        Including:
        Product Titles
        Quantities
        Subtotals
        Taxes
        Shipping Rate
        Shipping Info
        Billing Info
        Authorization Code

      • Special Requirements
        Should be an easily printable page.
        Only shows last 4 digits of credit card number.

      • Pre-Conditions
        - An order has been completed, authorized, and sent to fulfillment.

      • Post-Conditions
        - Users cart session is removed

      • Cross Reference
        -         Check Out Cart

      • Assumptions
        none

    9. Name: Sign On Admin

      • Actors:
        Admin

      • Brief Description
        This use case details how the Admin signs into the system.

      • Flow of Events
        Basic Flow
        User Action System Response
        1. User Selects “Sign On” 2. System Displays Interface for user to enter Username and Password.
        3. User Enters Username and Password and Submits. 4. System validates that the username and password are correct.
        5. Additional Auth Checks
        6. Systems Logs User into Admin Interface.
        7. Admin Session created.

        Alternative Flow
        User Action System Response
        4.1 Invalid Username or Password.
        4.2 System Displays interface to correct Username and password.
        4.3 User corrects and resubmits. 4.4 Return to 4.

      • Pre-Conditions
        Admin is setup in system.

      • Post-Conditions
        Admin is logged into the system and can make changes to the inventory and orders.

    11. Name: Manage Inventory

      • Actors:
        Admin

      • Brief Description
        This use case details how the Shopping Cart Admin can Add, Remove, and Edit items in the shopping cart.

      • Flow of Events
        Basic Flow
        User Action System Response
        1. User selects "Add Product" or "Update Product" 2. System Displays Interface to input
        Fields:
        -Title
        -Description
        -Price
        -Image
        -SKU
        -SalePrice
        -OnSale

        5. Systems writes item to database.

    12. Name: Admin Product Search

      • Actors:
        Admin

      • Brief Description
        Searches the backing store

      • Flow of Events
        Basic Flow
        User Action System Response
        1. Admin selects search for products 2. System Displays Interface to input search strings for Title, Description, and SKU. The interface also allows Admin to show products that are visible, not-visible or both. The interface also allows Admin to show products that are on sale, not-on sale, or both.
        3. User inputs string and submits. 4. Systems check products in the backing store for the exact, case insensitive search string.

        		5. System displays all results on one page. Each product displays the option to remove or edit or view the product.

        Alternative Flow
        User Action System Response
        5.1 System Found No Matches.
        5.2 System Tells the Admin that no products were found and allows them to change their search string and try again.

      • Pre-Conditions

      • Post-Conditions

      • Cross Reference
        Manage Inventory

      • Assumptions

      Top

        Basic Flow
        User Action System Response

      • Add Sub Category
      • Edit
      • Remove
      • Manage Products
      • Quick Product List
        • 3. Admin selects to "Add Sub Category" or "Edit" 4. The system displays the interface to add a Sub Category. Firlds are:
      • Name
      • Header
      • Visible
      • Image
        *** If the category is being edited, the fields are auto-poulated with the existing values
        		•
        5. Admin enters all nessecary info 6. System validates the information and saves in the backing store

        		7. System redirects back to the Manage Category main interface.

        Alternative Flow
        User Action System Response

        		 6.1 System could not validate the information

        		6.2 System redirects back to the "Add Sub Category" interface

        Alternative Flow
        User Action System Response
        3.1 Admin select "Remove" 3.2 The system asks for confirmation from the user
        3.3 Admin confirms 3.4 The system removes category and all subcategories.

        		3.5 System redirects back to the "Add Sub Category" interface
